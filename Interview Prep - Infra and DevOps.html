1. Design a CI/CD pipeline for a microservices architecture on Kubernetes.
💡 Hint: Cover GitOps, image scanning, helm charts, canary deployments, rollback strategies.

I would design a GitOps-based CI/CD pipeline leveraging our existing tool stack: GitHub Actions for CI, ArgoCD for GitOps-driven CD, Helm for packaging, and IKP/GKE as the target Kubernetes platform.

1. Source Control & Branching Strategy:

  • Adopt GitHub Flow or trunk-based development with protected main branches. Each microservice has its own repository with a standardized structure: /src, /helm, /docker, /.github/workflows.
  • Require pull requests with mandatory code review (minimum 2 approvers), status checks (build, test, security scan), and branch protection rules.

2. Continuous Integration (CI) - GitHub Actions:

  • Build Stage: Triggered on every PR and merge to main. Uses GitHub Actions runners to compile code, run unit tests, and generate artifacts.
  • Containerization: Build Docker images with multi-stage builds for minimal attack surface. Tag images with Git commit SHA and semantic version.
  • Security Scanning: Integrate Trivy/Clair for image vulnerability scanning, Snyk for dependency checks, and SonarQube for code quality. Fail the pipeline on critical vulnerabilities.
  • Artifact Registry: Push verified images to Google Artifact Registry (GAR) with immutable tags and signed attestations.

3. Continuous Deployment (CD) - ArgoCD (GitOps):

  • Maintain a separate GitOps repository (e.g., k8s-manifests) containing Helm charts and environment-specific values files (values-dev.yaml, values-staging.yaml, values-prod.yaml).
  • ArgoCD watches this repository and automatically syncs desired state to IKP/GKE clusters. This ensures declarative, version-controlled, and auditable deployments.
  • Use ArgoCD ApplicationSets for managing multiple microservices across environments with a single configuration.

4. Helm Chart Management:

  • Create a standardized Helm chart template (umbrella chart) for all microservices, including ConfigMaps, Secrets (external-secrets operator), Services, Ingress, and HorizontalPodAutoscaler (HPA).
  • Store common Helm charts in a private Helm repository (e.g., ChartMuseum or OCI registry) for reusability.

5. Progressive Delivery:

  • Implement canary deployments using Argo Rollouts or Flagger. Route a small percentage of traffic (e.g., 5%) to the new version, monitor key metrics (error rate, latency, custom business metrics) via Prometheus/Grafana.
  • Automated promotion: If metrics remain healthy for a defined period, gradually increase traffic to 100%. If anomalies detected, automatic rollback to the previous stable version.

6. Rollback Strategy:

  • Helm rollback: helm rollback <release> <revision> for immediate reversion to a known good state.
  • ArgoCD sync history: One-click rollback to any previous Git commit state.
  • Database migrations: Use Flyway/Liquibase with backward-compatible changes (expand-contract pattern) to ensure schema changes don't block rollbacks.

7. Secrets & Configuration:

  • Use External Secrets Operator to sync secrets from Google Secret Manager / Vault into Kubernetes, avoiding hardcoded credentials in Git.
  • Manage non-secret configuration via ConfigMaps, with environment-specific overrides in GitOps values files.

8. Notifications & Observability:

  • Integrate pipeline notifications with Slack/Teams for build success/failure, deployment status, and rollback events.
  • Correlate deployment events with application metrics in Grafana to quickly identify the impact of releases.

6. What are the key factors and design principles for high availability, especially regarding multi-site data centers, data synchronization, and service failover?
💡 Hint: Cover redundancy, data consistency models, CAP theorem, RTO/RPO, active-active vs active-passive, load balancing, health checks.

High availability is not just about uptime percentages; it's about designing systems that can withstand failures gracefully. My approach covers infrastructure redundancy, data synchronization, and intelligent service switching.

1. Infrastructure & Network Redundancy:

  • Multi-Site / Multi-Region Deployment: Architect applications across at least two geographically separated data centers (e.g., HK + mainland China) to mitigate regional disasters. Use GKE regional clusters or IKP multi-zone deployments to ensure node-level redundancy.
  • Network Redundancy: Implement redundant network paths (MPLS, VPN, dedicated interconnects) with automatic BGP failover. Use Global Load Balancers (e.g., GCP Global LB, F5) to route traffic based on health and proximity.
  • Eliminate Single Points of Failure (SPOF): Every component—from load balancers to database nodes—must have a hot standby or run in a clustered mode.

2. Data Synchronization & Consistency:

  • RPO & RTO Definitions: Before choosing a strategy, define Recovery Point Objective (how much data loss is acceptable) and Recovery Time Objective (how fast to recover). For core banking, we target RPO ≈ 0 and RTO < 60 seconds.
  • Synchronous vs. Asynchronous Replication:
    • Synchronous: Used for critical transactional data (e.g., Oracle Data Guard Max Availability, Cloud SQL HA). Guarantees zero data loss but introduces latency.
    • Asynchronous: Used for analytics and non-critical data (e.g., Cassandra cross-DC replication, Kafka MirrorMaker). Offers better performance but may have seconds of lag.
  • CAP Theorem Trade-offs: Banking systems typically favor Consistency + Partition Tolerance (CP) over Availability during network splits. For example, Oracle RAC ensures consistency, while Cassandra allows tunable consistency (LOCAL_QUORUM vs. QUORUM).
  • Conflict Resolution: For active-active setups, implement vector clocks, last-write-wins (LWW), or application-level reconciliation logic to handle split-brain scenarios.

3. Service Switching & Failover Mechanisms:

  • Active-Active vs. Active-Passive:
    • Active-Active: Both sites handle traffic simultaneously. Requires data bi-directional sync and conflict resolution. Best for read-heavy workloads.
    • Active-Passive: Primary site handles traffic; secondary is on standby. Simpler data flow but wastes resources. Common for legacy monolithic systems.
  • Health Checks & Auto-Failover: Implement multi-layer health checks (infrastructure, application, business logic). Use tools like ITRS/Geneos or cloud-native health checks to trigger automatic failover when thresholds are breached.
  • DNS & Traffic Management: Use GeoDNS or Global Server Load Balancing (GSLB) to route users to the nearest healthy site. Implement TTL adjustments for faster DNS propagation during failover.
  • Graceful Degradation: Design services to degrade functionality rather than fail completely. For example, if the recommendation engine is down, show cached or default content.

4. Application-Level Resilience:

  • Circuit Breakers: Prevent cascading failures by failing fast when downstream services are unhealthy (e.g., using Istio or Resilience4j).
  • Retry & Timeout Policies: Implement exponential backoff with jitter for transient failures. Set aggressive timeouts to avoid resource exhaustion.
  • Bulkheads: Isolate resources (e.g., thread pools, connection pools) so that a failure in one component doesn't exhaust shared resources.
  • Idempotency: Ensure that retried requests don't cause duplicate processing, critical for payment and settlement systems.

5. Testing & Validation:

  • Chaos Engineering: Regularly simulate failures (node crashes, network partitions, AZ outages) using tools like Chaos Monkey or Gremlin to validate resilience.
  • DR Drills: Conduct quarterly disaster recovery exercises involving full site failover, data restoration, and business continuity validation.
  • Game Days: Cross-functional exercises where teams practice incident response under simulated high-pressure scenarios.

6. Observability During Failures:

  • Ensure that monitoring and logging remain available during outages. Use out-of-band monitoring (separate infrastructure) to avoid blind spots when the primary system fails.
  • Implement runbooks and automated remediation scripts accessible from both sites.
